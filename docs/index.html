Vibe coding moves faster than memory.

Repo-Docs is not a file-tree tour or a chat transcript. It keeps a small project guide beside the source so the next reader can follow proof, not guesswork.

Without repo-docs
  • Code changed quickly, but the reason stayed in chat.
  • Files exist, yet no one can explain the real behavior path.
  • README, source, tests, and agent memory drift apart.
  • The next agent starts by rediscovering the same context.
With repo-docs
  • A walkthrough of one real run from entry to output.
  • Concept pages for the few ideas that actually matter.
  • Reference pages for commands, fields, schemas, and evidence.
  • A sync rule that keeps future answers tied to current source.
Observe first One real behavior before file inventory.
Evidence path Commands, contracts, and source links stay attached.
Minimal sync Only the stale page gets patched.

Evidence loop

Keep understanding tied to the path code actually took.

Observe one real behavior, name the concepts, anchor lookup in source evidence, then repair only the page that drifted. The loop keeps repo docs current without another full rewrite.

Observe Run a behavior through source, tests, data, or UI.
Name Give the stable concepts their own module pages.
Anchor Move exact lookup into references with source evidence.
Repair When code drifts, update the smallest owning page.

Skill modes

Four modes, one rule: prove before writing.

Repo-Docs changes shape with the state of the project, but it keeps the same evidence discipline each time.

What happened?

Build

First durable guide for a repo that already exists.

Sync

Repair stale understanding after code, config, data, or tests change.

Seed

Separate planned work, unknowns, and confirmed facts before code exists.

Refine

Adjust the guide when a repo question reveals the wrong reader model.

Sync loop

A small loop that keeps the guide current.

Read

Open the guide page that should answer the question.

Inspect

Check the current source, command, artifact, config, or data.

Patch

Update the owning page only when the gap is durable.

Answer

Reply from evidence and leave the repo easier to continue.
