## Understanding Brief

| Field | Guide answer |
| --- | --- |
| Scenario | A developer starts `aider` in a git repo, gives a coding request, and the default diff-style coder applies a SEARCH/REPLACE edit to a file in the chat. |
| Input | The observable input is CLI arguments and a user message. The package exposes the `aider` command through `aider.main:main` in [`pyproject.toml`](https://github.com/Aider-AI/aider/blob/5dc9490bb35f9729ef2c95d00a19ccd30c26339c/pyproject.toml), and `main()` accepts `argv` before parsing config and environment in [`aider/main.py`](https://github.com/Aider-AI/aider/blob/5dc9490bb35f9729ef2c95d00a19ccd30c26339c/aider/main.py). |
| Success / output | A file is changed, and when git auto-commit is enabled the edited file can be committed with an Aider-generated message. Tests assert changed file content and commit counts in [`tests/basic/test_coder.py`](https://github.com/Aider-AI/aider/blob/5dc9490bb35f9729ef2c95d00a19ccd30c26339c/tests/basic/test_coder.py). |
| Hard part | The path is not just "send prompt, write response." It must preserve the user-controlled file set, parse model output into an edit format, avoid clobbering ignored or unapproved files, preserve dirty work for undo, and turn lint/test failures back into model-visible feedback. |
| Boundary / failure | A malformed or non-matching SEARCH block raises a `ValueError`; `apply_updates()` records it as a malformed response and stores a reflected message instead of pretending the edit succeeded. |
| Falsifying check | If `tests/basic/test_editblock.py::TestEditBlockCoder::test_full_edit` stopped showing content changed from `one\ntwo\nthree\n` to `one\nnew\nthree\n`, or if `test_full_edit_dry_run` changed the file during dry run, this walkthrough would be wrong. |
| Likely reader follow-up | A careful reader will ask how Aider chooses the exact edit format or model. This first guide answers only the default edit lifecycle and points to source evidence for model/config selection; deeper model registry behavior is out of scope. |

## Coverage Notes

Covered: the terminal CLI startup path, `Coder` selection, a default diff edit, edit permission checks, dirty-file pre-commit, malformed edit reflection, lint/test reflection hooks, and auto-commit.

Checked but not traced end to end: GUI launch, watch mode, copy/paste mode, repo-map ranking internals, model metadata registration, OAuth onboarding, voice input, browser scraping, architect/editor split, and help search. These are adjacent surfaces that reuse or feed the `Coder` loop but deserve their own walkthroughs.
